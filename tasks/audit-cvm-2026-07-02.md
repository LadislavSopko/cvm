# Architecture & Foundations Audit — CVM

- **Project**: CVM (Cognitive Virtual Machine) — custom bytecode VM exposed as an MCP server, published to npm as `cvm-server@1.2.0`
- **Stack**: TypeScript 5.8, Node 18.16+, Nx 21.2 monorepo (7 packages + 1 app), Vitest, pino, MongoDB (optional), `@modelcontextprotocol/sdk`
- **Deployment / users**: single author + public npm (anyone can `npx cvm-server@latest`) → security weighted accordingly
- **Phase**: released, maintenance
- **Date**: 2026-07-02
- **Auditor**: Architecture Audit mindset (KISS/DRY/YAGNI + Foundations + Security)
- **Evidence tools**: LSAI (TypeScript/JavaScript plugins Ready), ripgrep/find scoped to `packages/*/src`, `apps/*/src`, `test/`; excluded `node_modules`, `dist`, `out-tsc`, `coverage`, `.cvm`.

> **Evidence-discipline corrections to Memory Bank claims** (deeper check overturned prior notes):
> - MB `techContext`: "zod … NOT USED despite available" → **corrected**: zod **is** used for every MCP tool input schema (`mcp-server.ts:82-538`). It is *not* used for env/config validation — that is the real gap.
> - MB: "@types/pino deprecated + unused" → **corrected**: pino **is** actively used (`types/logger.ts:4`); `@types/pino` ships as a *runtime* dep (a real defect, section 3/7), but it is not "unused".
> - MB: "nxCloudId … undeclared → 401" → **corrected**: `nxCloudId` **is** declared in `nx.json:20` but has **no access token** → orphan config (section 7).

## Executive Summary (IT)

CVM è un progetto **solido nel nucleo e sobrio nell'architettura**: la VM bytecode custom è una scelta *giustificata* (è il vero moat — esecuzione durevole e ripristinabile per un agente AI), non overengineering. La superficie di esecuzione è **pulita**: nessun `eval`, `child_process`, rete o shell nel codice VM; la sandbox del filesystem è default-deny e la fix del prefix-match (`a10b425`) è presente; gli input MCP sono validati con zod; le operazioni distruttive richiedono un `confirmToken`; il logging va su file (corretto per il transport stdio).

Le debolezze sono **di contorno, non di cuore**, e derivano dall'assenza di guardrail industriali: **nessuna CI, nessun lint**, l'app deployabile **non ha test**, la `config.execution` (limiti di tempo/stack/output) è **calcolata ma mai applicata** (config decorativa → gap DoS reale), l'accesso a `process.env` è **duplicato in 5 file**, c'è **version-drift di mongodb** (4 range diversi) e artefatti spuri committati. La sicurezza è buona per il contesto, con due rischi residui: nessun `realpath` (escape via symlink di directory intermedia) e stato vulnerabilità dipendenze **non misurabile** (`npm audit` bloccato dal proxy nexus, HTTP 400).

**Priorità**: (1) applicare davvero i limiti di esecuzione o rimuoverli; (2) aggiungere CI+lint minimale; (3) `realpath` nella sandbox; (4) centralizzare la config con zod; (5) pulire drift e clutter.

## 1. KISS Analysis

| Component | Complexity (1-5) | Justification | Simpler Alternative | Estimated Saving |
|---|---|---|---|---|
| Custom bytecode VM (parser→compiler→stack VM) | 4 | **Justified** — durable, resumable execution for an AI agent (zero-infra + survive-process-death). A simpler state machine cannot serialize/resume arbitrary control flow. | None that meets the requirement | — (keep) |
| Storage adapter pattern (file + mongodb) | 2 | Justified — two real backends behind one interface (`StorageAdapter`, `storage.ts:10`) | — | — (keep) |
| Opcode handler pattern (per-opcode handler) | 2 | Justified — testable, extensible (`handlers/types.ts:20`) | — | — (keep) |
| App version resolution (`main.ts:27-44`) | 3 | **Accidental** — hardcoded fallback `'0.4.3'` + 3-path package.json search + duplicate fallbacks `0.0.1`/`1.0.0` elsewhere | Read version once from bundled package.json; single constant | ~0.5 day, removes stale-version confusion |
| `config.execution` limits (`config.ts:52-54`) | 3 | **Accidental** — computed but never passed to VM (`main.ts:63` passes only `version`) | Enforce them, or delete the block | see §3/§4 |
| Logger typed `any` + `require('pino')` + Proxy (`logger.ts`) | 2 | **Accidental** — CommonJS `require` hack + `any` in a `nodenext` strict project | `import pino from 'pino'`, type the export | 0.5 day, restores type safety |

**Verdict**: Core is KISS-compliant. Accidental complexity is confined to app glue (version resolution, dead config, logger typing). **Score 4/5.**

## 3. YAGNI Analysis

- 🔴 **`config.execution` (maxExecutionTime / maxStackSize / maxOutputSize)** — computed by `loadConfig()` but **never wired** to the VM (`new CVMMcpServer(version)` receives only version; grep shows the fields appear *only* in `config.ts`). Dead config that *looks* like a safety limit but enforces nothing → both YAGNI and a security gap (§6, A04). **Either enforce or remove.**
- 🟡 **`packages/mongodb`** — MB records it as DEAD (no imports; `storage` has its own `mongodb-adapter.ts`). It still pulls `mongodb ^6.3.0` into the graph. Confirm and remove, or fold into `storage`.
- 🟡 **`config.api` (port/host) commented block** (`config.ts:25-29`) — speculative future feature; delete until needed.
- 🟡 **Nx Cloud** — `nxCloudId` declared with no token and no CI to use it (§7). Distributed cache is YAGNI for a single-author repo.
- 🟢 **Custom VM, storage adapters, opcode handlers** — justified by current requirements. Keep.

**Verdict**: A few decorative/dead constructs; nothing structurally speculative. **Score 3/5.**

## 4. Overengineering Analysis

- **Infra vs need**: minimal — a published npm package + optional local MongoDB. No cloud runtime cost, no HA, no service mesh. Appropriate.
- **Architecture vs team (1 dev)**: the layered monorepo (parser→types→storage→vm→mcp-server→app) is understandable and debuggable by one person. Not over-decomposed.
- **Process vs speed**: currently *under*-processed (no CI/lint), the opposite of overengineering.
- **Stack usage**: TypeScript compiler API used purposefully for parsing CVM source. `zod` used at the right boundary (MCP tools). No tech used at 10%.
- **One residual smell**: `test/programs/tddab/planexecutor.ts` is a **production runtime artifact** (shipped as builtin `@planexecutor`, copied into `dist/programs/` by `vite.config.ts:34`, published in the tarball) but lives under `test/` — misleading layout, not overengineering.

**Verdict**: Not overengineered; if anything under-tooled on process. **Score 4/5.**

## 6. Security Analysis

Threat model: MCP **stdio** transport (local process, no network listener) → no remote auth surface. The real attack surface is (a) arbitrary CVM programs executed by the VM and (b) supply chain of the published npm package.

### 6.1 OWASP Top 10

| Risk | Status | Evidence |
|---|---|---|
| A01 Broken Access Control | 🟢 OK | stdio = local trust boundary; destructive tools (`delete_program`/`delete_execution`) gated by `confirmToken` (`mcp-server.ts:413,492`). |
| A02 Cryptographic Failures | 🟢 OK | No crypto claims; no secrets transmitted. Local dev mongo creds only (`root:example`). |
| A03 Injection | 🟢 OK | No `eval`/`new Function`/`child_process`/shell in VM (`rg` → only `require('pino')` in logger). MCP inputs validated via zod. VM is bytecode-interpreted, not `eval`-based. |
| A04 Insecure Design | 🟡 Risk | **Execution limits (`maxExecutionTime`/`maxStackSize`/`maxOutputSize`) are configured but never enforced** — a malicious/buggy CVM program can loop/allocate unbounded. |
| A05 Security Misconfiguration | 🟡 Risk | FS sandbox **default-deny** (good, `file-system.ts:51`), but relies on env being set; dead `config.execution` gives a false sense of limits; docker uses throwaway creds + wrong `container_name: prontuari-mongo`. |
| A06 Vulnerable Components | 🟡 Risk (**unmeasured**) | `npm audit` **blocked** by nexus proxy (HTTP 400) → advisory status cannot be determined. SDK installed 1.17.2 (upstream ~1.29). Must be measured against public registry. |
| A07 Auth Failures | 🟢 OK (N/A) | No network auth surface (stdio). |
| A08 Data Integrity Failures | 🟢 OK | State serialized as JSON to local file/mongo; no deserialization of untrusted code. |
| A09 Logging & Monitoring | 🟢 OK | pino file logging present; errors persisted in VM state for inspection. |
| A10 SSRF | 🟢 OK | CVM programs have **no network access** by design. |

### 6.2 Secrets Management
- `.env` is **not tracked** (`git ls-files .env` → empty) → good. Contains only a local dev mongo credential (`root:example`).
- No secrets in source (hits for "token"/"secret" are lexer tokens in `parser/`, not credentials).
- `docker/docker-compose.yml` embeds `MONGO_INITDB_ROOT_PASSWORD: example` (throwaway, local only) and a copy-pasted `container_name: prontuari-mongo` from another project — clean up.

### 6.3 Input Validation
- MCP boundary: **every** `server.tool()` declares a zod shape (`mcp-server.ts:82-538`) → validated by the SDK. Good. Minor: `source`/`result` are unbounded `z.string()` (no size cap) → couples with the missing output/exec limits (A04).

### 6.4 Sandbox (filesystem)
- **Default-deny** when no `CVM_SANDBOX_PATHS`/`CVM_SANDBOX_ROOT` (`file-system.ts:51-53`). ✓
- Prefix-match escape **fixed** (`normalizedPath === sandboxPath || startsWith(sandboxPath + path.sep)`, `:57-60`, commit `a10b425`). ✓
- Symlinks: final-component symlinks blocked in read/write/list (`lstatSync`, `:153,:187,:121`). ✓
- **Residual 🟡**: no `fs.realpathSync` — an **intermediate directory symlink** inside the sandbox pointing outside is not caught (only the final path component is `lstat`-checked; `isPathAllowed` resolves lexically, not physically). Requires an external actor to plant a symlink in the sandbox, so severity is moderate; fix with `realpath` for defense-in-depth. The `normalizedPath.includes('..')` check (`:46`) is effectively dead after `path.resolve` (harmless).

### 6.5 Dependency Vulnerabilities
- **Cannot be verified** here — `npm audit` returns nexus 400. Run against the public registry (`npm audit --registry https://registry.npmjs.org/`) in CI.

### 6.6 Transport Security
- stdio only; no HTTPS/CORS surface. N/A.

**Verdict**: Good posture for the context (clean execution surface, default-deny sandbox, validated inputs, gated destructive ops). Fix: enforce execution limits (A04), add `realpath` (sandbox), measure `npm audit` (A06). **Score 4/5.**

## 8. Final Recommendations (by ROI)

### Quick wins (< 1 day)
1. **Wire or delete `config.execution`** *(High, A04)* — pass limits into `VMManager`/VM and enforce (step budget, stack cap, output cap), or remove the dead block. Closes the DoS gap and the YAGNI smell.
2. **`fs.realpathSync` in `isPathAllowed`** *(Risk)* — resolve physical path before the boundary check; kills intermediate-symlink escape. Remove the dead `includes('..')`.
3. **Fix published deps** *(Medium)* — move `@types/pino`/`pino-pretty` to devDependencies; reconsider shipping `typescript`.
4. **Remove tracked junk** *(Low)* — `git rm --cached counter.ts graph.html tsconfig.tsbuildinfo`; add `*.tsbuildinfo` to `.gitignore`.
5. **Measure `npm audit`** *(Risk, A06)* — run against `registry.npmjs.org` and record the result.

### Medium term (1-2 weeks)
6. **Minimal CI + ESLint** *(High, #1)* — GitHub Actions: `npm ci` → `nx run-many typecheck,test` → `eslint --max-warnings 0`. Add `noUnusedParameters`. This alone lifts several Partials.
7. **Centralize config with zod** *(High, #8)* — one validated schema in the app; inject values into storage/logger/file-system instead of each reading `process.env`. NaN-guard the numeric limits.
8. **Pin mongodb version** *(High, #2)* — one floor across `types`/`mongodb`/`storage`/`cvm-server`; resolve the two-lockfile split.
9. **Test the app** *(Medium, #13)* — spec `config.ts`/`main.ts` wiring (currently 0 tests on the deployable).
10. **Relocate `planexecutor.ts`** — move from `test/programs/tddab/` to `apps/cvm-server/programs/`; update `vite.config.ts`.

### Strategic (1-3 months)
11. **Slim the core** *(#10)* — remove `mongodb`/`pino` from `packages/types` deps (pass logger in; keep only contracts) so the "core" is truly zero-dep.
12. **Immutability pass** *(#6)* — `readonly` on shared DTOs / `Readonly<T>` on collections.
13. **Confirm & drop dead `packages/mongodb`** if unused (MB flags it DEAD).
