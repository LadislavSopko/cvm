### Addendum — `@planexecutor` read in full (2026-06-09, post-Protocol-D)

The KISS verdict below was challenged with "is the custom bytecode VM invented complexity? Could a state machine + JSON do it?" To answer honestly I read the production builtin `test/programs/tddab/planexecutor.ts` (302 LOC, single `main()`) in full. Findings, evidence-based:

1. **The VM is NOT obviously replaceable.** The program needs three properties simultaneously: zero infrastructure, survival of full process death mid-workflow, and driving an external MCP client. Generator/continuation JS dies with the process (fails property 2). Temporal / Azure durable-functions provide exactly this durability but require a server/runtime (fails property 1). No off-the-shelf substrate is plainly simpler while keeping all three. **My initial "invented complexity" suspicion was wrong; the KISS score stands at 4/5.**

2. **But the "instruction-level resumability" moat is overstated.** Evidence — planexecutor does NOT rely on the VM's per-instruction state serialization for real durability. It writes its own **block-level** checkpoint:
   ```
   var progressRaw = fs.readFile(".cvm/uplan-progress.json");          // resume at BLOCK granularity
   fs.writeFile(".cvm/uplan-progress.json", JSON.stringify(completedBlocks));  // manual checkpoint per block
   ```
   The VM's `pc`/`stack`/`variables` serialization handles the *clean-pause* case (resume at the exact `CC`). The durability that actually matters — recovering from a dirty crash mid-block — is done by the application writing JSON by hand, exactly as a trivial state machine would. **The VM does the easy part elaborately; the hard part is done by hand on top of it.** The real differentiator is therefore "durable execution with zero infra driving an MCP agent", not the instruction granularity per se. The Value Assessment at the end of this report is corrected accordingly.

- 🟢 **Keep** — storage adapter (both backends actually used), handler pattern, sandboxed FS, TDDAB plan parser + `@planexecutor` builtin (this is the actual product surface used daily), dual licensing scaffolding.

## 4. Overengineering Analysis

**Infrastructure vs real requirements:** essentially zero infrastructure — a stdio MCP server installed via npx. Correctly sized. MongoDB is opt-in, file storage default. No premature HA, no premature scaling. ✅

**Architecture vs team size:** one person can hold the whole system; the GPS/CC() paradigm is documented well (README, docs/, memory-bank). The VM is the only specialized-knowledge area, and it is the product itself. ✅

**Process vs speed:** no CI at all (see Foundations #1) — the opposite of overengineering; this is *under*-engineering of process. Local `nx run-many -t typecheck/test` works and passes, but nothing enforces it.

**Technology stack:** every dependency in the published app is used (typescript = compiler API for the parser; zod = MCP tool schemas; pino = logging; mongodb = optional backend) **except** `@types/pino`. MCP SDK declared `^1.12.2`, installed 1.17.2, upstream is at **1.29.0**; code uses the deprecated `server.tool()` registration API (current SDK favors `registerTool`). Not broken today, but drift on the one protocol-critical dependency deserves a scheduled bump.

**Overall:** the system is refreshingly boring where it should be and complex only in its core. No overengineering verdict; the real risk is process under-engineering (CI, lint).

## 6. Security Analysis

**Threat model:** local stdio MCP server, single user, no network listener, no auth surface. Most OWASP web categories do not apply; the relevant surfaces are (a) the VM sandbox, (b) the file system service, (c) supply chain.

**6.1 OWASP-mapped findings:**

| Risk | Status | Evidence |
|---|---|---|
| A01 Broken Access Control | 🟡 **Risk** | **Sandbox prefix-match bypass**: `file-system.ts:56-58` — `normalizedPath.startsWith(sandboxPath)` without a trailing separator. With `CVM_SANDBOX_ROOT=/data/sandbox`, the path `/data/sandbox-evil/x` passes the check. Fix: compare against `sandboxPath + path.sep` (or `path.relative` not starting with `..`). The `includes('..')` check at line 46 is dead after `path.resolve` |
| A03 Injection | 🟢 OK | Programs are compiled to bytecode and interpreted — **no `eval`, no `new Function`** (verified via grep across packages). The TS compiler API only parses; MongoDB access uses driver methods, not string queries |
| A05 Misconfiguration | 🟡 Risk | `CVM_STORAGE_TYPE` unvalidated cast (config.ts); deny-by-default sandbox when unconfigured ✅ (good) |
| A06 Vulnerable Components | 🟡 **Risk (unknown)** | `npm audit` **cannot run**: the Nexus group proxy returns HTTP 400 on the audit endpoint. There is no CI audit either. Vulnerability exposure is currently *unmeasured*. MCP SDK 1.17.2 vs upstream 1.29.0 |
| A08 Data Integrity | 🟢 OK | State persisted as JSON files or Mongo docs, local trust boundary |
| A09 Logging | 🟢 OK | pino file logging; failures logged; no secrets logged (connection string never printed) |
| A02/A04/A07/A10 | 🟢 N/A-OK | No crypto, no auth surface, no outbound fetch in VM (no network opcodes — by design) |

**6.2 Secrets:** none in code (grep clean — only `connectionString` constructor parameters); `.env` gitignored ✅; `.env.example` provided ✅. The `~/.npmrc` auth token is outside the repo ✅.

**6.3/6.4:** Input validation at MCP boundary is done with zod schemas per tool ✅. No auth needed (stdio).

**6.5 Positive security properties worth stating:** deny-by-default sandbox, symlink skip on list/read/write, no-information-leak null returns, no dynamic code execution. For a VM that an AI drives, this is a sound posture — the one real flaw is the prefix-match bug.

## Value Assessment (requested in audit scope)

**What CVM actually is:** a deterministic bytecode VM for a TypeScript subset, where `CC("prompt")` is an *opcode that suspends the machine* and delegates one cognitive step to an AI over MCP. State (variables, heap, program counter) lives in the VM, not in the model's context. This inverts the usual "code calls AI" into "program drives AI" — closer to Temporal-style durable execution than to agent frameworks.

**Genuine differentiators (verified in code, not marketing):**
1. **Durable execution with zero infrastructure, driving an MCP agent** — execution state is serialized so a crashed/restarted Claude session resumes a long workflow. LangGraph/CrewAI checkpoint at graph-node granularity and live in Python processes; Temporal/durable-functions give the same durability but need a server. CVM gives it via `npx` with no runtime. *(Corrected: the VM's per-instruction serialization handles clean pauses, but the real crash-recovery durability in the flagship `@planexecutor` is done by an application-level block checkpoint — see Section 2 Addendum. The moat is the zero-infra durable-execution-for-an-agent combination, not the instruction granularity by itself.)*
2. **Determinism with a human veto** — control flow, loops, and counters are computed, never hallucinated; the verdict-gate contract (passed/failed tokens) keeps the AI's output machine-checkable.
3. **Zero-infrastructure delivery** — `npx cvm-server` into any MCP client; competitors need a Python runtime, an orchestrator, or a SaaS.
4. **The TDDAB plan executor** is the killer app: it turns the VM from a curiosity into a working autonomous-TDD harness, and it is dogfooded daily on CVM itself.

**Honest market read:** ~1,750 downloads/month after the first year is a real but niche signal. Headwinds: AGPL scares enterprise evaluation (mitigated by the commercial license, but only if anyone reaches the negotiation step); the language subset (no user functions yet) limits program complexity; the category ("durable execution for AI cognition") doesn't have a search term yet; bus factor = 1. Tailwinds: the agent-orchestration market is moving exactly toward externalized state and resumable workflows; MCP adoption is growing; CVM was early (June 2025) with a coherent, tested implementation.

**Verdict:** highest near-term value is as a *productivity multiplier for its own author's workflow* (which it demonstrably is — the j-* skills and planexecutor pipeline run on it) and as a *credibility asset* for the dual-license/consulting channel. As a standalone commercial product it needs the function-support milestone, CI/security hardening (this audit), and one public case study of a long-running plan executed end-to-end. The engineering substance is real; the moat is the resumability granularity, and no mainstream framework currently replicates it.
